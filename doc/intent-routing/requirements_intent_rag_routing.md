# 需求文档：A/B/C 意图路由与多向量库检索增强

## 1. 背景
当前 Web 路由主要是问答类（知识库）与非问答类（直接 loop）两段流程，无法满足以下目标：
- 运维操作场景先基于工具语义检索，再进入 loop 执行。
- 排障场景先基于 skills 语义检索，再进入 loop 执行。
- 与知识库检索隔离，避免不同语料混用。

## 2. 目标
实现三类意图路由（A/B/C）与三套独立向量库：
- A（知识问答）：走知识库向量库，直接返回（保持现有行为）。
- B（运维操作）：走工具向量库检索，取 top2，作为系统补充上下文进入 loop。
- C（故障排障）：走 skills 向量库检索，取 top2，作为系统补充上下文进入 loop。

## 3. 最终确认的需求约束
1. 意图分类固定为 A/B/C。  
2. C 仅查询 skills 向量库。  
3. 工具数据源为 ToolRegistry 已注册工具（不是全代码静态扫描）。  
4. MCP 工具来源为静态配置（`config.mcp.servers`）。  
5. 检索结果作为系统补充上下文注入（不拼到用户输入）。  
6. 工具库、skills 库、知识库使用不同 `chroma_dir`。  
7. 检索策略：`top2 + 不 rerank`（工具和 skills 都如此）。

## 4. 范围
### 4.1 In Scope
- Web 侧意图路由 A/B/C 的代码逻辑与分发。
- 新增工具索引初始化逻辑（本地注册工具 + MCP 静态工具）。
- 新增 skills 索引初始化逻辑（workspace/builtin skills）。
- 新增工具库与 skills 库专用 Chroma 存储目录。
- B/C 路由中检索 top2 并注入系统补充上下文后进入 loop。

### 4.2 Out of Scope
- rerank 能力与阈值路由（本期不做）。
- 动态 MCP 工具发现（本期仅静态配置）。
- 知识库 A 类流程重构（保持原行为）。

## 5. 业务流程要求
### 5.1 A 类（知识问答）
用户输入 -> 意图识别 A -> 查询知识库向量库 -> 返回结果（维持现有）。

### 5.2 B 类（运维操作）
用户输入 -> 意图识别 B -> 查询工具向量库（top2） -> 生成系统补充上下文 -> 进入 loop。

### 5.3 C 类（故障排障）
用户输入 -> 意图识别 C -> 查询 skills 向量库（top2） -> 生成系统补充上下文 -> 进入 loop。

## 6. 非功能需求
- 性能：B/C 检索阶段只取 top2，不做 rerank，控制整体时延。
- 可维护性：工具库、skills 库、知识库三库隔离，避免语料污染。
- 可观测性：记录意图类别、检索库名、top2 命中条目、注入上下文长度。

## 7. 验收标准
1. A/B/C 能稳定识别并进入对应分支（代码与提示词一致）。  
2. B 分支只访问工具库；C 分支只访问 skills 库；A 分支只访问知识库。  
3. B/C 分支检索数量固定 top2，且未触发 rerank。  
4. B/C 分支进入 loop 时可见系统补充上下文注入。  
5. `chroma_dir` 至少存在三套隔离目录，互不复用。  

## 8. 风险与注意事项
- 若 ToolRegistry 注册工具较少，B 类召回可能不足，需要先确保工具注册完整。
- 若 skills 文本过长，建议初始化时做切块，避免检索噪音。
- 若意图识别返回值校验未同步 A/B/C，C 会误降级到 A（需修正）。
